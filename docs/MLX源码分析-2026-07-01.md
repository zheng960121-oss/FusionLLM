## 2. 6 个核心模块

### 2.1 runner.go —— 入口和模型加载

**关键代码段**（mlxrunner/runner.go）：

```go
type Runner struct {
    Model         base.Model
    Tokenizer     *tokenizer.Tokenizer
    Requests      chan Request
    Sampler       *sample.Sampler
    cache         kvCache
    contextLength int
    mlxThread     *mlxthread.Thread
    spec          *speculation  // nil when no draft head
}

func (r *Runner) Load(modelName string) error {
    root, err := model.Open(modelName)  // 打开 model manifest
    m, err := base.New(root)            // 解析 config.json, 找到 arch
    
    tensors, err := loadTensorsFromManifest(root)  // 读 safetensors
    m.LoadWeights(tensors)                          // 分配到 nn.Embedding / nn.Linear
    
    // draft model: 独立或 SelfDraft 嵌入
    draftModel := base.NewDraft(root, m) || m.(SelfDraft).SelfDraft()
    
    // 关键：Pin 所有 weights
    collected := mlx.Collect(m)  // 收集所有 mlx.Array
    for _, arr := range collected { mlx.Pin(arr) }
    mlx.Sweep()
    mlx.Eval(collected...)
    
    r.Model = m
    r.spec = newSpeculation(r, draftModel)
    mlx.EnableCompile()  // 启用 MLX graph compile 缓存
}
```

**关键观察**：
- `mlx.Pin(arr)` —— 把 weights 锁在内存里（不被 sweep 回收），对应我们 llama.cpp 的 mlock
- `mlx.Collect(m)` —— 通过反射收集 Model struct 里所有 `*mlx.Array` 字段
- `mlx.Sweep()` + `mlx.Eval(...)` —— 立即求值，释放中间 buffer
- `mlx.EnableCompile()` —— 启用 graph compile，相同 shape 的 forward 复用编译结果

### 2.2 mlxthread —— 单线程串行化（精妙设计）

**关键设计**（x/internal/mlxthread/thread.go）：

```go
type Thread struct {
    name    string
    jobs    chan job       // 序列化队列
    done    chan struct{}
    stopping atomic.Bool
}

// Start 创建一个长期 worker goroutine 锁定到单一 OS 线程
func Start(name string, init func() error) (*Thread, error) {
    t := &Thread{jobs: make(chan job), done: make(chan struct{})}
    go t.loop(init, initResult)  // worker 在单 OS 线程上跑
    return t, nil
}

func (t *Thread) Do(ctx context.Context, fn func() error) error {
    res, err := t.enqueue(ctx, fn, false, false)
    return res.err
}
```

**为什么单线程？**：
- MLX 的 lazy evaluation 模型要求严格串行（一次只能有一个 compute graph 在飞）
- Apple Silicon UMA（统一内存）已经免了 CPU↔GPU 拷贝
- 单线程避免 MLX 内部锁竞争
- 通过 `runtime.LockOSThread()` 绑定到固定线程，CPU cache 更友好

**对比我们 llama.cpp fork**：没有这种结构，llama.cpp 本身就是单线程推理 + 批处理，所以不需要。

### 2.3 server.go —— HTTP API

**路由**（mlxrunner/server.go）：


**核心循环**：

```go
func (r *Runner) Run(host, port string, mux http.Handler) error {
    g, ctx := errgroup.WithContext(ctx)
    g.Go(func() error {
        for {
            select {
            case request := <-r.Requests:  // 串行处理请求
                r.runRequest(request)
            }
        }
    })
    g.Go(func() error { return http.ListenAndServe(...) })

```

**关键观察**：
- **Ollama mlxrunner 接受的是 /v1/completions 端点，不是 OpenAI 兼容的 /v1/chat/completions**
- 输入：JSON with `prompt` + `options` (temperature, top_p, etc.)
- 输出：streaming JSONL
- 没有 chat template 处理（Ollama main 做）
- `mlx.CheckInit()` + `mlx.GPUIsAvailable()` 决定走 GPU 还是 CPU

### 2.4 cache.go + cache_trie.go —— Paged KV Cache（核心创新）

**数据结构**（cache.go）：

```go
const maxPagedOutBytes int64 = 8 << 30  // 8 GiB eviction threshold

type kvCache struct {
    root          *trieNode         // prefix trie root
    activePath    []*trieNode       // 当前 root→leaf 路径
    caches        []cache.Cache     // 每层一个 cache
    pagedOutBytes int64             // 总 paged out 字节数（用于 LRU）
}

type trieNode struct {
    tokens    []int32              // 压缩边（多 token 共享一个 node）
    endOffset int                  // 累积 token 数
    parent    *trieNode
    children  []*trieNode
    lastUsed  time.Time            // LRU 戳
    snapshots []cache.Snapshot     // 每层 paged-out 数据
    user      bool                 // true = 显式 restore point
}
```

**关键操作**：

1. **`begin(m, inputs)`** —— 找最长前缀匹配
   - 遍历 trie 找最匹配 node
   - 至少留 1 token 让 pipeline seed
   - 返回 cacheSession（含 caches, inputs, remaining）

2. **`switchToPath(newPath, matched)`** —— 切换分支
   - page-out 旧 leaf 节点的 snapshot（如果不是 ancestor）
   - 跳过 ancestor 节点
   - page-in 新路径上的 snapshot
   - 所有 cache 对齐到 minCacheOffset
   - `enforceEvictionPolicy()` 触发 LRU

3. **`enforceEvictionPolicy()`** —— LRU 回收
   - 遍历所有 nodes（不只 active path）
   - 累计 pagedOutBytes
   - 超 8 GiB 阈值时，按 lastUsed 排序，丢弃最旧

4. **Prefill 时 schedule snapshots**：
   ```go
   const snapshotInterval = 8192
   for offset := snapshotInterval; offset < len(inputs); offset += snapshotInterval {
       snapshotOffsets = append(snapshotOffsets, offset)
   }
   const preThinking = 4
   if end := len(inputs) - preThinking; end > 0 {
       snapshotOffsets = append(snapshotOffsets, end)
   }
   session.schedulePrefillSnapshots(snapshotOffsets)
   ```

**关键观察**：
- **8 GiB SSD/host paged 阈值** —— 默认 8 GB，超了就 LRU
- **每 8192 token 一个 snapshot** —— 恢复粒度
- **prefix 4 token 前再加 snapshot** —— "思考前"恢复点
- **压缩 trie** —— 兄弟节点不共享 token 前缀（节省内存）

**对比我们的 llama.cpp fork**：
- 我们只有 L0 (GPU) + L1 (CPU mlock) + L2 (SSD mmap)
- 没有 prefix trie（每次重新分配）
- 没有 LRU 跨请求（每次请求独立 cache）
- 没有 page-in / page-out 协议

**对 16GB Air 的实际收益**：
- Prefix trie 在多轮对话场景下大幅省 memory + 时间
- LRU 防止 SSD 写满
- 但单请求单 session 时，这些功能用不上

### 2.5 speculate.go + mtp.go —— Speculative Decode

**顶层**（speculate.go）：

```go
type speculation struct {
    r     *Runner
    draft base.DraftModel
    caches  []cache.Cache    // 完整 cache 切片
    draftKV []cache.Cache    // drafter 自己的 cache slots
    targets []cache.Cache    // target cache slots（被 drafter 写）
    depth *depthController   // 学习 cost/acceptance
}

type speculationSession struct {
    s       *speculation
    drafter drafter
    enabled bool
    limit   int  // 当前 draft length
    stats   specStats
}
```

**MTP drafter**（mtp.go）：

```go
type mtpDrafter struct {
    spec              *speculation
    frontier          int
    frontierHidden    *mlx.Array  // target 最后 hidden
    committedDraftOffset int       // drafter 已写入的 slot
    pendingTokens     []*mlx.Array
    pendingHiddens    []*mlx.Array
    pendingCount      int
    heldHidden        *mlx.Array
    heldProjected     *mlx.Array
}
```

**Drafter 接口**（基础 model/base/base.go）：

```go
type DraftModel interface {
    Draft(b *batch.Batch, caches []cache.Cache) (hidden, projected *mlx.Array)
    Unembed(x *mlx.Array) *mlx.Array
    DraftCaches(caches []cache.Cache) []cache.Cache  // 选 drafter 自己的 slot
    LoadWeights(tensors map[string]*mlx.Array) error
}

type SelfDraft interface {
    SelfDraft() DraftModel  // 内嵌在 target weights 里
}
```

**Decode 循环**（speculativeDecoder.next）：

```go
func (st *speculativeDecoder) next(remaining int) ([]sampler.Result, error) {
    s.beginRound()
    var candidates *draftCandidates
    if s.limit > 0 {
        candidates = s.drafter.propose(st.current.Token, min(s.limit, remaining-1))
    }
    var accepted, observed int
    if candidates == nil {
        results, err = st.park(remaining)  // 普通 decode
    } else {
        results, accepted, observed, err = st.s.accept(...)
    }
    s.endRound(drafted, accepted, observed)
    return results, nil
}
```

**Depth Controller**（跨请求学习）：

```go
// 每次 endRound 调用：
s.spec.depth.acc.observe(observed, accepted)  // 记录 acceptance rate
s.limit = s.spec.depth.next()                  // 决定下次的 draft length
```

**关键观察**：
- **Drafter 写自己的 draftKV + 读 target caches** —— 不重写 target
- **跨请求持续学习 depth** —— 多轮后 draft length 收敛到最优
- **EAGLE-style 边界处理** —— 注释里提到 "TODO: EAGLE keeps draft one slot behind"
- **mxfp4 等量化支持** —— 通过 `mlx.Collect` + `mlx.Pin`
- **Park 模式** —— 当不能 draft 时（logprobs 等），继续维护 draft cache

**对比我们的 llama.cpp fork**：
- 我们只有 DSpark 一条 spec decode 路径
- Ollama 的实现：通用 drafter 接口 + depth controller
- Ollama 的更通用（MTP、EAGLE、单步 head 都支持）

### 2.6 x/models/qwen3/qwen3.go —— 模型实现

**Config**：

```go
type Config struct {
    HiddenSize            int32
    NumHiddenLayers       int32
    IntermediateSize      int32
    NumAttentionHeads     int32
    NumKeyValueHeads      int32  // GQA 支持
    HeadDim               int32
    MaxPositionEmbeddings int32
    // ... 量化参数
}
```

**Forward**（核心 kernel）：

```go
func (m *Model) Forward(b *batch.Batch, caches []cache.Cache) *mlx.Array {
    h := m.EmbedTokens.Forward(b.InputIDs)
    for i, layer := range m.Layers {
        var c cache.Cache
        if caches != nil && i < len(caches) { c = caches[i] }
        h = layer.Forward(h, b, c, positions, B, L, m.Config)
    }
    return m.Norm.Forward(h, m.RMSNormEps)
}

func (a *Attention) Forward(x, b, c, positions, B, L, cfg) *mlx.Array {
    q := a.QProj.Forward(x); k := a.KProj.Forward(x); v := a.VProj.Forward(x)
    q = Reshape(q, B, L, NumHeads, HeadDim)
    k = Reshape(k, B, L, NumKVHeads, HeadDim)  // GQA: K/V heads 少
    v = Reshape(v, B, L, NumKVHeads, HeadDim)
    q = a.QNorm.Forward(q, eps); k = a.KNorm.Forward(k, eps)
    q = Transpose(q, 0, 2, 1, 3); k = Transpose(k, 0, 2, 1, 3); v = Transpose(v, 0, 2, 1, 3)
    q = mlx.RoPEWithBase(q, head_dim, traditional=false, theta, scale, positions)
    k = mlx.RoPEWithBase(k, head_dim, traditional=false, theta, scale, positions)
    
    var kv nn.SDPAOption
    if c != nil {
        history := c.(cache.Attention).Update(b, k, v)  // 更新 KV cache
        kv = nn.WithKVHistory(history)  // 用历史 KV 计算 attention
    } else {
        kv = nn.WithKV(k, v, b.SeqQueryLens)
    }
    out := nn.ScaledDotProductAttention(b, q, cfg.Scale, kv, nn.WithMask(nn.CausalMask()))
    return a.OProj.Forward(Reshape(Transpose(out, 0, 2, 1, 3), B, L, NumHeads*HeadDim))
}
```

**关键观察**：
- **MLX SDPA 原生支持 GQA**（注释：avoid materializing repeated K/V tensors）
- **Flash Attention 内置在 SDPA**（MLX 的 SDPA 是 fused kernel）
- **RoPE 在 heads 之后做**（标准做法）
- **Q/K norm 一起做**（Qwen3 特色）
- **CausalMask 是 lazy SDPA option**（不构造 full mask）

**这就是为什么 14B 32K 在 MLX 上能跑**：
- MLX 的 SDPA 是 flash attention fused kernel
- 不物化 O(n²) attention matrix
- 跟 llama.cpp 的 -fa on 等价
- Qwen2.5 → Qwen3 移植简单（架构差异小）

## 3. 16GB Air "Gate" 真假验证

**之前的假设**：Ollama 把 MLX gate 在 32GB+ Mac
**实际**：

我搜了所有代码，**没找到 hard-coded 32GB check**。但有一些相关检查：

1. `loadImageModel()` in `x/imagegen/imagegen.go`：
   ```go
   availableMemory := mlx.GetMemoryLimit()
   if availableMemory > 0 && requiredMemory > 0 && availableMemory < requiredMemory {
       return fmt.Errorf("insufficient memory for image generation: need %d GB, have %d GB", ...)
   }
   ```
   这只对 image generation 生效。

2. 模型层 mlxrunner 没有 explicit memory gate。

**实际原因**（综合分析）：
- Ollama MLX 首个支持的模型是 **Qwen3.5-35B-A3B**（MoE，激活 3B）
- 35B Q4 ≈ 20 GB
- 16GB Air 装不下 35B
- 所以 16GB Air 实际跑不到 MLX
- **其他模型没 MLX port**（Qwen2.5 缺）
- **结果**：16GB Air 在 Ollama 0.30 仍用 llama.cpp Metal

**含义**：

如果我们能：
1. **写一个 Qwen2.5 MLX port**（Qwen3 → Qwen2.5 移植 1-2 周）
2. **PR 给 Ollama 上游**（风险：可能被拒或慢）
3. **自己 fork Ollama**（维护负担重）

或：

**用 mlx-lm 直接绕开 Ollama**（Qwen2.5 已经有，1-2 周出 demo）

## 4. mlx-lm vs Ollama MLX runner 详细对比

| 维度 | Ollama MLX runner | mlx-lm (Apple 官方) |
|:---|:---:|:---:|
| **核心语言** | Go + CGO | Python |
| **代码量** | ~5K 行 + MLX C++ | ~3K 行 |
| **Qwen2.5 支持** | ❌（需自己 port）| ✅（qwen2.py）|
| **16GB Air 跑 14B 32K** | ❌（无 Qwen2.5 port）| ✅ 待实测 |
| **Paged KV Cache** | ✅ 8 GiB + prefix trie + LRU | ⚠️ ChunkedKVCache（无 trie）|
| **LRU Prompt Cache** | ✅ 跨请求复用 | ✅ LRUPromptCache |
| **Quantized KV** | ✅（推测，cache.Snapshot 抽象）| ✅ QuantizedKVCache |
| **Spec Decode** | ✅ MTP drafter + depth controller | ✅ Draft model 通用接口 |
| **Speculative draft 类型** | MTP, EAGLE, single-position | 通用 draft model |
| **HTTP API** | /v1/completions（不是 OpenAI 兼容）| OpenAI 兼容 |
| **Multi-session 优化** | ✅ Prefix trie | ⚠️ 需自己实现 |
| **二次开发难度** | 🟡 中（Go + CGO）| 🟢 低（Python）|
| **社区支持** | Ollama 团队 | Apple 团队 |
| **生产稳定性** | ✅ Ollama 0.30 stable | ✅ mlx-lm stable |
| **生态集成** | Ollama 全套 | mlx 生态 |

**核心差异**：
1. **Ollama MLX runner 更工业级**（Prefix trie 跨请求、HTTP server、生产 stable）
2. **mlx-lm 更灵活**（Python、Qwen2.5 原生、容易 fork 改）

## 5. 对我们项目的具体含义

### 5.1 mlx-lm 路径（推荐）

**做法**：
1. `pip install mlx-lm`
2. 写薄 Python 包装：
   ```python
   from mlx_lm import load, generate
   from mlx_lm.models.cache import LRUPromptCache, ChunkedKVCache
   from mlx_lm.generate import speculative_generate_step
   from mlx_lm.server import main as serve
   
   class FusionLLM:
       def __init__(self, model_path):
           self.model, self.tokenizer = load(model_path)
           self.cache = LRUPromptCache()  # 跨请求 LRU
       
       def prefill(self, tokens):
           cache = make_prompt_cache(self.model, max_kv_size=8192)  # ChunkedKVCache
           return cache
       
       def generate(self, prompt, draft_model=None):
           return generate(self.model, self.tokenizer, prompt=prompt,
                          draft_model=draft_model, num_draft_tokens=4)
   ```
3. 加 FusionLLM 特色：
   - 显式 mlock（用 `mlx.set_wired_limit()`）
   - 跨 session KV cache 持久化
   - DSpark draft model 集成（用 `speculative_generate_step`）
4. 包装为 OpenAI 兼容 HTTP server

**时间**：1-2 周
**性能**：推测 14B 32K 9-12 t/s gen（+30-80% vs llama.cpp）

## 7. 关键洞察

1. **MLX 推理不需要 fork 巨型项目** —— mlx-lm 3K 行 Python 已经覆盖 90% 需求
2. **Ollama MLX runner 的核心创新是 prefix trie + LRU paged** —— 但单 session 16GB Air 跑 14B 32K 用不到
3. **16GB "gate" 不是 hard code** —— 是模型选择（35B 不装 16GB）的副作用
4. **Qwen2.5 → Qwen3 移植不难**（架构差异小）—— 但维护负担重
5. **spec decode 在 14B 32K 上增益有限**（gen 6.7 t/s 是 bandwidth-bound，不是 compute-bound）
