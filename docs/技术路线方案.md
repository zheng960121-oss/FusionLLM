## 0. 一页纸项目定位

**FusionLLM = 在 Apple Silicon MacBook 上跑 70B 级大模型的本地推理引擎**

| 维度 | 决策 |
|---|---|
| 目标硬件 | **M5 MacBook Air（16GB 统一内存）= 唯一目标 + 验证平台** |
| 目标模型 | 70B Q4_K_M 量化（约 40GB 权重，依赖 SSD offload）|
| 目标场景 | **单流交互式推理**（不是批处理吞吐量）|
| 优化指标 | first-token latency ≤ 2s，per-token latency ≤ 80ms |
| **终极目标（路径 c）** | **70B Q4 + 32K-128K 上下文跑通** |
| 真正差异化 | **KV Cache 分层（GPU/CPU/SSD）+ mlock 显式生命周期管理** |
| 非差异化（借用现有）| 滑动窗口预取、零拷贝 Metal、Paged KV Cache、GGUF 解析 |

**项目本质**：工程整合项目，把 PowerInfer-2 + LMCache + llama.cpp paged cache 的核心思想系统性搬到 Apple Silicon 上。不是研究突破。

## 1. 关键技术决策（动手前必须敲定）

### D1：底层框架选择

| 方案 | 优势 | 风险 |
|---|---|---|
| **A. Fork llama.cpp** ✅ | Metal backend 现成、GGUF 解析现成、paged KV cache 现成、社区大 | 改 ggml 内部结构难度大 |
| B. 基于 MLX | 内存管理更现代、API 友好 | 性能调优经验少、社区小 |
| C. 从头写 | 完全控制 | 工作量巨大、重复造轮子 |

**结论：选 A**。理由：真正新东西（mlock 生命周期、KV Cache 分层）都可以在 llama.cpp 上层做 driver，不必动 ggml 核心。

### D2：模型格式

- **A. 自研 .fusion 格式**（原方案想法）
- **B. 直接支持 GGUF**，调度器做在上层 ✅

**结论：选 B**。A 让用户重新下载/转换模型，生态门槛太高；B 可以直接用 Ollama / HuggingFace 上的 GGUF 模型，零迁移成本。

### D3：调度器架构

| 方案 | 特点 | 适用 |
|---|---|---|
| A. 全局 look-ahead（FlexGen 风格）| 适合批处理 | 不适用本目标 |
| **B. 滑动窗口 + 显式 mlock**（原方案设计）| 适合单流 | ✅ |
| C. Reactive prefetch | 实现简单 | 性能不可预测 |

**结论：选 B**。窗口大小需要可调（启动时根据模型大小自动选）。

### D4：I/O 路径

- ~~MTLIOCommandQueue~~（Apple Silicon 上基本不可用）
- **dispatch_io_t + pthread**（推荐）
- aio / IOCommandQueue（备选）

## 3. Phase 0：风险关门（1-2 周，**不写产品代码**）

**这是整个项目最重要的阶段**。在确认所有未知数之前，一行产品代码都不写。

### PoC-1：GPU 缺页行为（1-2 天，**M5 MacBook Air 直接跑**）

**目标**：确定 Metal 访问 mlock 失效页的后果。

```
步骤：
1. mmap 一个 4GB 稀疏文件（fallocate 或 truncate + sparse）
2. mlock 前 1GB 物理驻留，后 3GB 不 mlock
3. 创建 MTLBuffer_A（包 mlock 区 1GB）+ MTLBuffer_B（包非 mlock 区 1GB）
4. 提交 kernel：A 写 0xAA，B 写 0xBB
5. GPU 命令完成后 sleep 2 秒
6. munlock + madvise(DONTNEED) Buffer A 的物理页（mmap 保留）
7. 再提交 kernel：用 Buffer A 做读
8. 观察：panic？timeout？静默错？数据正确？
```

**Go 判据**：M4 Pro 上能跑且数据正确 → 进入 Phase 1。
**No-go 判据**：崩溃或静默错 → **项目暂停重新评估**（M5 大概率不会变好）。
**M5 真机到手后**：重做一次，看 Apple 是否有改进。

### PoC-2：mlock RLIMIT 验证（半天）

```
1. ulimit -l（看默认值，通常几 MB）
2. 写小测试程序：mlock 1GB
3. 尝试 mlock 8GB
4. 验证 entitlements 路径（com.apple.developer.kernel.increased-memory-limit）
```

**Go 判据**：能用 entitlements 解决。
**No-go 判据**：必须 root 或改系统 limit（不可分发）。

### PoC-3：macOS MADV_DONTNEED 行为（半天）

```
1. mmap 1GB
2. mlock，memset 0xAA
3. 立即 madvise(DONTNEED)
4. 再访问那段地址，看触发缺页 + 重新从文件读
5. 测量物理内存是否真的释放（top / vmmap）
```

### PoC-4：KV Cache SSD 读写循环（路径 c 专用，1-2 天，**Phase 2 之前必做**）

**目标**：验证 KV Cache 落 SSD + 重新加载的可行性，是 Phase 3 的提前风险验证。

```
步骤：
1. 生成模拟 KV Cache 结构（伪随机数据，1GB）
2. 序列化为 SSD 文件
3. mmap 该文件
4. 创建 MTLBuffer 包装 mmap 区
5. kernel 读 + 验证数据一致
6. 模拟换出：madvise(DONTNEED)
7. 模拟换入：访问触发的页
8. 测量：读写带宽、延迟、内存占用峰值
9. 验证：多次换入换出后数据完整性
```

**Go 判据**：1GB KV Cache 换入换出 < 200ms，10 次循环无数据损坏。
**No-go 判据**：换入 > 500ms 或数据损坏 → KV Cache 分层方案需重新设计（例如改为预拷贝 + 完整 evict）。

**Phase 0 同步阅读任务**：
- [ ] PowerInfer-2 论文，1 页摘要
- [ ] llama.cpp paged KV cache 的 PR / commit，1 页笔记
- [ ] LMCache GitHub 源码，重点看 KV 换入换出策略

**Phase 0 Gate**：所有 PoC 通过 + 阅读笔记完成 → 进 Phase 1。

## 4. 分阶段开发路线

### Phase 1：最小可用推理（2-3 周）

**目标**：在 M4 Pro 上跑 7B Q4 模型，对比原版 llama.cpp 性能。

```
1. Fork llama.cpp，建立 fusion-llm 分支
2. 加一个 --fusion-driver 编译选项
3. 实现 mmap + mlock + madvise 的权重生命周期 hook
4. 跑 7B Q4，对比原 llama.cpp 的 first-token / per-token latency
5. 性能回归测试（不能比原版慢超过 5%）
```

**Gate**：7B 模型跑通，mlock 路径性能合理（不能比原版差太多）。

### Phase 2：滑动窗口预取（3-4 周）

**目标**：跑 13B Q4 模型，验证预取效果。

```
1. 实现"窗口"概念：当前层 + 预取 N 层
2. 实现调度器：计算完一层后，触发下一批 I/O
3. 实现 I/O 线程（pthread + dispatch_io_t）
4. 实现 Metal 完成回调 → mlock 状态机
5. 调窗口大小（实测最佳值，建议 4-8 层）
```

**Gate**：13B 模型 per-token latency 稳定，I/O stall < 10%。

### Phase 3：KV Cache 分层（**P0 差异化**）（4-6 周）

**目标**：跑 70B Q4 + 4K 上下文，验证 KV Cache 落 SSD。

```
1. 集成 llama.cpp paged KV cache 源码
2. 实现 KV block 热度统计（LRU + access counter）
3. 实现 KV block 序列化到 SSD（独立文件，不混模型权重）
4. 实现换入换出：内存压力 → 冷 block 落盘；需要时 → 异步换入
5. 实现 mmap 化 KV cache（避免来回 malloc）
6. 跑 70B + 4K 上下文，测最大并发 session 数
```

**Gate**：70B Q4 + 4K 上下文跑通，单 session per-token latency ≤ 100ms。

### Phase 4：生态对接（2-3 周）

```
1. 兼容 Ollama REST API（/api/generate, /api/chat）
2. 兼容 Modelfile（简化版）
3. CLI：fusionllm run, fusionllm pull, fusionllm ps
4. 文档：使用手册 + 性能调优指南
5. 简单 TUI（可选，hamel-style）
```

**Gate**：能用 Ollama 客户端无修改对接。

### Phase 5：长上下文优化（**已合并入 Phase 3，路径 c 模式**）

路径 c 下 Phase 5 不独立存在，合并进 Phase 3：

```
合并到 Phase 3：
1. activation memory 优化（gradient checkpointing 的推理版）
2. 注意力分块（chunked attention）
3. 实测 70B + 32K-128K 上下文
```

## 6. 失败预案

### 修正后的内存管理设计（关键，基于 PoC-1 Test 4 实测）

**原设计**：mlock（保留）+ madvise（释放）双机制精确控制
**实测发现**：macOS 的 madvise 几乎从不真正释放页（即使 6GB 压力下也不释放）
**修正设计**：**只用 mlock/munlock** 作为控制手段

```
活跃滑动窗口 (6 layers × 500MB = 3GB): mlock'd
  → 永远驻留物理内存
  → GPU zero-copy 访问
  → 推理延迟 < 1ms

其余模型层: mmap'd 但 NOT mlock'd
  → 依赖 OS file cache
  → 真实内存压力下自动换出
  → 冷层访问触发 SSD I/O
```

**为什么这个设计更好**：
- mlock 是 OS 强制保证的（不会失效）
- 不依赖 madvise 的不可预测行为
- 简化调度逻辑：只需要决定哪些层 mlock
- 16GB 设备：3GB mlock + ~10GB file cache + 3GB 系统 ≈ 16GB

### 如果 PoC-1（GPU 缺页）失败

- **退路 1**：放弃 mlock 路径，改用"权重预拷贝到 Metal buffer"（牺牲零拷贝）
- **退路 2**：缩小范围到 13B-30B 模型（24GB 够装权重）
- **退路 3**：放弃 FusionLLM，做个 Ollama + MLX 的配置优化工具

### 如果 Phase 3（KV Cache 分层）性能不达标

- **退路 1**：把 KV Cache 分层作为可选功能（关闭后回到传统模式）
- **退路 2**：只在 64GB+ 设备启用分层，24GB 只用基础滑动窗口

### 如果 M5 真机行为与 M4 差异大

- 暂停主线，启动 M5-specific PoC 重做
- 预留 2-4 周 buffer

## 附录 B：术语速查

| 术语 | 含义 |
|---|---|
| mmap | 内存映射文件，虚拟地址 ↔ 文件偏移 |
| mlock | 锁定物理页，禁止换出 |
| madvise(DONTNEED) | 建议内核释放指定范围的物理页 |
| MTLResourceStorageModeShared | Metal 共享存储模式，CPU/GPU 共享同一份内存 |
| dispatch_io_t | GCD 的异步 I/O 抽象 |
| Prefill | 推理首阶段，处理整个 prompt |
| Decode | 推理后续阶段，逐 token 生成 |
| KV Cache | 推理中缓存 K/V 矩阵，避免重复计算 |
| paged KV cache | 把 KV cache 分成固定大小 block 管理（vLLM 首创）|
| Hot/Cold tiering | 把热数据留在快存储，冷数据放慢存储 |
