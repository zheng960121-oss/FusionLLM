# MLX Runner 集成研究

**日期**：2026-07-01
**触发**：老大 08:52 拍板"研究一下如何将MLX runner集成"
**研究者**：小兰 (MiniMax-M3)

---

## 1. 核心结论（30 秒看完）

**不要集成 Ollama 的 MLX runner。改用 mlx-lm（Apple 官方 Python 库）。**

| 维度 | Ollama MLX runner | mlx-lm (Apple 官方) |
|:---|:---:|:---:|
| 16GB Air 可用 | ❌ 被 gate 在 ≥32GB | ✅ 无内存 gate |
| Qwen2.5 支持 | ❌ 只有 Qwen3/3.5 | ✅ Qwen2/2.5/3/3.5 全支持 |
| KV cache 功能 | ✅ Paged + Prefix trie | ✅ Chunked + Quantized + Rotating + LRU |
| Speculative decoding | ✅ MTP drafter | ✅ Draft model 通用接口 |
| 安装难度 | 🔴 集成到 Ollama 巨型 Go 服务 | 🟢 `pip install mlx-lm` |
| 二次开发 | 🟡 Go + MLX CGO | 🟢 Python |
| M5 Neural Engine | ✅ 已用 | ✅ 通过 MLX 框架 |

**Ollama MLX runner 3 个 blocker**：
1. **32GB+ 内存 gate**（Ollama 0.30 官方限制，16GB Air 拿不到）
2. **Qwen2.5 不在 MLX 模型列表**（只有 Qwen3/3.5/3.5-MoE，Qwen2.5-14B 走 llama.cpp Metal）
3. **集成成本高**（Ollama 100K+ LOC Go，绕开 gate + 写 Qwen2.5 MLX port = 2-3 月）

**mlx-lm 完美对应**：
- 无 32GB gate
- Qwen2.5-14B 原生支持（`mlx_lm/models/qwen2.py`）
- 所有 KV tier + spec decode 功能都有
- Python 二次开发简单

## 2. Ollama MLX runner 源码分析

### 2.1 模型注册（Ollama mlxrunner）

源码：`ollama/ollama/x/mlxrunner/imports.go`

```go
import (
    _ "github.com/ollama/ollama/x/models/cohere2_moe"
    _ "github.com/ollama/ollama/x/models/gemma3"
    _ "github.com/ollama/ollama/x/models/gemma4"
    _ "github.com/ollama/ollama/x/models/glm4_moe_lite"
    _ "github.com/ollama/ollama/x/models/laguna"
    _ "github.com/ollama/ollama/x/models/llama"
    _ "github.com/ollama/ollama/x/models/qwen3"
    _ "github.com/ollama/ollama/x/models/qwen3_5"
    _ "github.com/ollama/ollama/x/models/qwen3_5_moe"
)
```

**❌ 没有 Qwen2 / Qwen2.5**。即使我们绕过 32GB gate，Qwen2.5-14B 也不能跑 Ollama MLX。

### 2.2 32GB 内存 gate（Ollama 0.30）

来自 Ollama 官方 blog（runaihome.com 2026 报道）：
> "Nothing in Ollama 0.19 changes the situation for sub-32GB Macs. The Metal backend is unchanged and still the default when MLX prerequisites aren't met."

源码位置（partial，rate-limited）：`server/sched.go` 的 `req.model.IsMLX()` 路径，gate 在 model resolution 阶段，未到 scheduler。

**绕过方式**（理论）：改 Go 代码 hard-code IsMLX() = true for Qwen2.5 + 改 32GB 检查为 16GB。**实际工作量**：1-2 周 fork + 长期维护负担。

### 2.3 Ollama Modelfile 不暴露 cache_type / mlock / flash_attn

读了 `docs/modelfile.mdx`，Modelfile 只支持：
- `num_ctx` (context size)
- `num_gpu` (auto, via -ngl auto)
- `num_predict` / `draft_num_predict` (spec decode)
- 标准 sampling params (temp, top_p, top_k, etc.)

**❌ 不支持**：`cache_type_k` / `cache_type_v` / `mlock` / `flash_attn` / `num_thread`

意味着即使 Modelfile 包装，也控制不了 14B 32K OOM 的关键 flags。

### 2.4 Ollama mlxrunner 已有功能（如果用得到）

- ✅ Paged KV cache (8 GiB SSD/host paged snapshot) — `cache.go`
- ✅ Prefix trie 共享 — `cache_trie.go`
- ✅ Speculative decoding (MTP drafter) — `speculate.go` + `mtp.go`
- ✅ Apple Silicon native (MLX)
- ✅ M5 Neural Engine 支持

但**对我们 16GB Air + Qwen2.5-14B 用不上**。

## 3. mlx-lm 全面对比

**仓库**：`ml-explore/mlx-lm`（Apple 官方）
**安装**：`pip install mlx-lm`
**大小**：~5K LOC Python（vs Ollama 100K+ LOC Go）

### 3.1 已有功能（Python 模块）

| 功能 | 模块 | 性能 |
|:---|:---|:---|
| **KV cache 类型** | `mlx_lm/models/cache.py` | |
| — 默认 KVCache | ✓ | 标准 |
| — RotatingKVCache | ✓ | 滑动窗口 |
| — **ChunkedKVCache** | ✓ | 分块（类似 paged）|
| — **QuantizedKVCache** | ✓ | KV 量化（q4_0/q8_0）|
| — BatchKVCache / BatchRotatingKVCache | ✓ | 批处理 |
| — **LRUPromptCache** | ✓ | Prompt 缓存复用 |
| **Speculative decoding** | `mlx_lm/generate.py` | |
| — Draft model 接口 | ✓ | 通用 |
| — `--draft-model` flag | ✓ | CLI 暴露 |
| — `--num-draft-tokens` | ✓ | CLI 暴露 |
| **HTTP server** | `mlx_lm/server.py` | |
| — OpenAI 兼容 | ✓ | |
| — chat/completions | ✓ | |
| — 流式响应 | ✓ | |
| **模型支持** | `mlx_lm/models/` | |
| — Qwen2 (Qwen2.5) | ✓ `qwen2.py` | **我们的目标** |
| — Qwen3 | ✓ `qwen3.py` | |
| — Qwen3.5 | ✓ `qwen3_5.py` | |
| — Llama / Gemma / etc. | ✓ | 完整 |

### 3.2 mlx_lm CLI 命令

```bash
# 加载 MLX 量化模型（mlx-community 在 HF 上有几千个 4-bit 模型）
mlx_lm.generate --model mlx-community/Qwen2.5-14B-Instruct-4bit --prompt "..."
mlx_lm.chat --model mlx-community/Qwen2.5-14B-Instruct-4bit
mlx_lm.server --model mlx-community/Qwen2.5-14B-Instruct-4bit
mlx_lm.cache_prompt --model ...   # 预计算 prompt cache
mlx_lm.benchmark --model ...
mlx_lm.quantize --model ...       # 量化 GGUF 到 MLX
```

### 3.3 mlx-lm 16GB Air 14B 32K 可行性

**理论**：
- 14B Q4 ~ 8.4 GB
- + mlx overhead ~1 GB
- + ChunkedKVCache 不物化 attention matrix
- + QuantizedKVCache 减少 KV memory
- 总 ~10-11 GB → 装得下 16GB Air

**未验证**（需要实测）。mlx-lm 0.20+ 文档说支持 16GB Mac。

**MLX 的 spec decode**（用 DSpark draft model）：
- mlx_lm spec decode 通过 `--draft-model` + `--num-draft-tokens`
- 我们已经有 DSpark GGUF writer（Phase 6 S9），可以输出 DSpark draft 到 MLX 格式
- mlx-lm 直接吃 HF safetensors 格式，需要做 DSpark → MLX 转换

### 3.4 性能预期

根据 groundy.com 2026 报告：
- M2 Ultra 192GB Qwen2.5 MLX 230 t/s vs llama.cpp 150 t/s = **+53%**
- M1 Max 32GB Qwen2.5-7B MLX 63.7 t/s vs llama.cpp 40.75 t/s = **+56%**

我们 14B 在 M5 Air 16GB 上：
- llama.cpp + FA + KV q4_0 Phase 4 实测 14B 32K = 6.7 t/s gen
- 推测 MLX 上 14B 32K = **9-12 t/s gen**（+30-80%）
- 14B 4K 短 prompt：MLX 可能 +50% 更快

**注意**：这个推测基于 groundy.com 数据，他们测的是短 context（未确认 32K 长 context 下 MLX 是否仍领先）。

## 6. 给老大的 3 条具体建议

### 建议 1：先实测 mlx-lm 14B 32K（1-2 小时，$0）

不做大决定，先跑测试。如果：
- ✅ 跑通 + ≥ 8 t/s gen → mlx-lm 路径成立，转方案 A
- ❌ OOM 或慢 → 退到 llama.cpp + Ollama 集成（路径 A 原计划）
- 🟡 跑通但慢（5-7 t/s）→ 比较 trade-off 再决定

### 建议 2：Ollama MLX runner 集成**不推荐**（1 月工作，3 blocker）

Ollama 100K+ LOC Go + 32GB gate + Qwen2.5 缺 = 投入产出比极差。

如果老大坚持要 Ollama 集成（基于用户/生态考量），最低成本是：
- 走 llama.cpp Metal（不碰 MLX）
- 写 Modelfile + 1-2 行 shell wrapper
- 跟 Ollama 默认体验一样
- **承认 6.7 t/s gen 是 16GB Air 物理上限**

### 建议 3：路径对比

| 方案 | 时间 | 14B 32K gen | 用户体验 | 投入 |
|:---|:---:|:---:|:---:|:---:|
| **mlx-lm 包装** | 1-2 周 | **9-12 t/s（推测）** | OpenAI 兼容 API + pip install | 🟢 低 |
| Ollama MLX 集成 | 2-3 月 | 6.7 t/s（被 gate 限制） | 跟 Ollama 默认一样 | 🔴 高 |
| llama.cpp fork + Ollama | 1 周 | 6.7 t/s | 跟 Ollama 默认一样 | 🟢 最低 |
| mlx-lm fork + DSpark spec | 4-6 周 | 12-25 t/s（spec 加速） | OpenAI 兼容 + 真正快 | 🟡 中 |
