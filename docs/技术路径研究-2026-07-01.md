## 1. 战略形势（必读）

### 1.1 Ollama 2026 H1 重大事件

| 日期 | 事件 | 对 FusionLLM 的冲击 |
|:---:|:---|:---|
| 2026-03-31 | **Ollama 0.19** 推出 MLX runner (preview) | 🔴 直接撞车 |
| 2026-04 | Ollama MLX 1.6x prefill / 2x gen 基准公布 | 🔴 性能对比劣势 |
| 2026-05-13 | **Ollama 0.30** MLX stable + llama.cpp 共存 | 🔴 赛道被占 |

**Ollama MLX runner 已经实现**（读源码确认）：

1. **Paged KV Cache**：8 GiB SSD/host paged snapshot，`cache.go` + `cache_trie.go` + `cacheSession`
2. **Prefix Trie 共享**：多会话复用 KV 缓存，`cache_trie.go`
3. **Speculative Decoding**：MTP drafter 接口 + DraftModel registry，`speculate.go` + `mtp.go` + `base.DraftModel`
4. **Apple Silicon Native**：MLX C++ + Go orchestration，UMA 优化
5. **M5 Neural Accelerator 支持**（Ollama 0.19 提到）

**Ollama 0.30 模型覆盖**：Qwen3.5-35B-A3B（首个 MLX 加速），其他（Llama 4 / Mistral Small 4 / Phi-4 / Gemma 3）还在 llama.cpp Metal。

### 1.2 Ollama MLX 在 16GB Air 上的可用性

**关键限制**（从 Ollama 官方 blog 确认）：

- **MLX 仅 ≥32GB Mac 启用**（M2 Ultra / M4 Max 32+）
- **16GB MacBook Air 仍跑 llama.cpp Metal** —— 我们的硬件
- **Ollama 16GB Air 7B-13B 实测 25-40 t/s**（MLX 不可用，跟我们 vanilla 14B 6.7 t/s 量级一致）

**含义**：Ollama 的 MLX 优势在 16GB Air 上拿不到。Ollama 0.30 也不会让 16GB Air 跑得更快（MLX 仍被 gate）。

### 1.3 16GB Air 14B 32K 性能预期

| 来源 | Prefill t/s | Gen t/s | 备注 |
|:---:|:---:|:---:|---|
| **Phase 4 实测**（vanilla FA + KV q4_0） | **~95** | **~6.7** | M5 Air 16GB |
| Ollama 0.30 llama.cpp 推测 | ~95 | ~6.7 | 同硬件同 backend |
| 14B 64K 预期 | OOM | OOM | 注意力 64K² = 64 GB，物理不可行 |
| 70B + 32K 任何 16GB Mac | 不可能 | 不可能 | 模型就装不下 |

**6.7 t/s gen 是 16GB Air 14B 32K 的物理上限**（M5 Metal bandwidth ~200 GB/s，14B matmul 每 token 约 30 GB 读）。

## 2. 4 条候选技术路径

### 2.1 路径 A：Ollama 集成（产品化）

**做什么**：把 FusionLLM 变成一个 Ollama Modelfile + runner patch，让用户 `ollama run qwen2.5-14b-fusion:32k` 就能用。

**实施**：

1. **Modelfile**（Ollama 自己的 model config 格式）
   ```
   FROM ~/Models/Qwen2.5-14B-Instruct-Q4_K_M.gguf
   PARAMETER num_ctx 32768
   PARAMETER num_gpu 99
   PARAMETER cache_type_k q4_0
   PARAMETER cache_type_v q4_0
   PARAMETER flash_attn true
   PARAMETER mlock true
   SYSTEM "..."
   ```
2. **README + 安装脚本**：3 行 shell（curl model + `ollama create` + `ollama run`）
3. **可选**：patch llama.cpp 默认参数（让 32K 也能默认开 FA）

**优势**：
- 1-2 周可上线 demo
- 真用户能直接用
- 跟 Ollama 生态兼容（用 Ollama API 的工具都能用）
- 借 Ollama 流量

**劣势**：
- 14B 32K 在 16GB Air 6.7 t/s，体感慢
- 跟"原始" Ollama 差异小（都是 llama.cpp + FA + KV q4_0）
- 没有 spec decode（Ollama MLX 才有，但 16GB Air 拿不到）
- 护城河薄

**工作量**：7-10 天

**价值评估**：
- **用户价值**：🟢 高（turnkey 体验）
- **技术护城河**：🟡 中（任何人都能 fork 一个类似 Modelfile）
- **学习价值**：🟢 高（建立 Ollama 集成 know-how）
- **收入潜力**：🟢 中（可做 Mac 16GB Air 用户的 LLM 工具）

---

### 2.2 路径 B：继续 llama.cpp fork 深度优化

**做什么**：把 FusionLLM 的 KV tier + sliding window + spec decode 都打磨好，作为 llama.cpp 的 upstream patch 提交。

**实施**：

1. **W2 sliding window 收尾**（sliding window 已实现 60%）
2. **DSpark spec decode 真训练**（~$240 云 GPU）
3. **KV tier 性能 bug 修复**（16K -33% 那个）
4. **打 upstream PR**：llama.cpp main 分支
5. **CI + 性能回归测试**

**优势**：
- 真实护城河（spec decode + sliding window + mlock 在 vanilla 之上）
- 6x spec decode 加速（如果 acceptance > 0.5）能让 6.7 t/s → 13-20 t/s
- 跟 Ollama MLX 路径差异化（spec decode 我们做的是 general llama.cpp，Ollama 的是 MLX-specific）
- 真技术深度

**劣势**：
- **DSpark 训练可能失败**（acceptance 还是 0，等云 GPU 跑一次才知道）
- 即使 acceptance > 0.5，spec decode 在 M5 Air 16GB 上也救不了 KV 装不下的问题
- 6-8 周工作量
- 需要云 GPU（$240 一次 + 时间）
- Upstream PR 可能被拒（llama.cpp 维护者不一定收）

**工作量**：6-8 周

**价值评估**：
- **用户价值**：🟡 中（spec decode 真能加速但 16GB Air 上限还是 6.7 t/s 量级）
- **技术护城河**：🟢 高（spec decode + sliding window 不容易被复制）
- **学习价值**：🟢 很高（spec decode 训练 + llama.cpp 源码深入）
- **收入潜力**：🟡 中（卖点是真性能，不一定有人付钱）

---

### 2.3 路径 C：pivot 到 MLX 实现

**做什么**：抛弃 llama.cpp fork，用 MLX 框架从零重写 spec decode + sliding window。

**实施**：

1. **学 MLX 框架**（Python + 少量 C++，Apple 官方）
2. **port Qwen2.5-14B 架构**到 MLX（参考 Ollama mlxrunner/model）
3. **port spec decode + sliding window**到 MLX
4. **M5 Neural Accelerator 利用**（M5 才有，Ollama 还没全用上）
5. **打包为 Python 库**（跟 Ollama MLX runner 平级竞争）

**优势**：
- MLX 性能 2x gen vs llama.cpp（如果 16GB Air 能用 MLX）
- 跟 Ollama 在 16GB Air 上一较高下
- M5 Neural Engine 是新硬件（2025 才有），差异化潜力

**劣势**：
- **Ollama 已经做了 MLX + paged cache + spec decode**，我们从 0 追
- Ollama 0.19 到现在 0.30 才 3 个月，但 35B 模型已经支持，生态压力大
- 16GB Air 上 MLX 还没启用（Ollama gate 在 32GB+），我们能不能绕开这个 gate ？
- MLX 模型覆盖窄（新模型先有 GGUF 再有 MLX）
- Python + MLX 慢于 C++（MLX 是 JIT，但 runtime overhead 大）
- 3-4 个月工作量
- 风险大

**工作量**：3-4 个月

**价值评估**：
- **用户价值**：🟡 中（如果能做 16GB Air MLX，会比 vanilla 快，但 Ollama 可能也加上）
- **技术护城河**：🟡 中（Ollama 在前面）
- **学习价值**：🟢 很高（深度 Apple 生态）
- **收入潜力**：🟡 中

---

### 2.4 路径 D：pivot 到 70B 64GB Mac Studio 方向

**做什么**：放弃 16GB Air 赛道，专门做 Mac Studio 64GB 的 70B 32K。

**实施**：

1. **买/借一台 64GB Mac Studio**（$2-3K）
2. **重写 KV tier + sliding window**针对大模型
3. **真 DSpark 训练**（用云 GPU 8×A100）
4. **Ollama Modelfile**（70B on Mac Studio 64GB）

**优势**：
- 真护城河：64GB Mac 上 70B 32K 没人做
- DSpark spec decode 在 70B 上收益最大（Ollama 0.30 还没真做 70B spec decode）
- 老大原 70B 梦想的实现

**劣势**：
- **64GB Mac Studio 硬件要花钱**（$2-3K + 时间）
- 8×A100 云训练 $240
- 跟 Ollama 0.30 35B-A3B 间接竞争（35B 也能 long context）
- 1-2 个月 + 硬件成本
- 老大可能不想花钱

**工作量**：1-2 个月 + $2.3K 硬件

**价值评估**：
- **用户价值**：🟡 中（70B on Mac Studio 64GB 用户群小）
- **技术护城河**：🟢 高（专属方向）
- **学习价值**：🟢 很高
- **收入潜力**：🟡 中

## 4. 我的推荐

**先 A（Ollama 集成）拿到 demo，再决定 B/C/D。**

理由：

1. **A 风险最低、见效最快**（1-2 周出 demo）—— 拿 demo 给老大看 / 给用户用，立刻验证产品形态
2. **A 跟 B/C/D 不互斥** —— A 的 Modelfile + install 脚本可以包装 B 的 spec decode 或 C 的 MLX 实现
3. **A 暴露真问题** —— 跑通 demo 才知道用户要什么（gen 速度？interface？model variety？），避免过早投入 B/C/D
4. **A 不烧钱** —— 0 成本，$0 硬件
5. **A 给老大一个 deadline** —— 1-2 周后 demo 出来，强制评估下一步

**A 之后的选择**：

- 如果 demo 用户要"更快" → 走 B（spec decode + sliding window）
- 如果 demo 用户要"更多模型" → 走 C（MLX 覆盖）
- 如果 demo 用户要"70B" → 走 D（买硬件）

**我的反对选项**：

- **不建议先走 B**：6-8 周太长，且 DSpark acceptance 还可能为 0
- **不建议先走 C**：Ollama 已经在 MLX 上，3-4 个月追不上
- **不建议先走 D**：$2-3K 硬件成本，1-2 个月时间，没有用户验证就投入

| 7-10 | 发布 + 推广 | GitHub release + HN/Reddit 帖 |

### 5.2 关键技术点

1. **Ollama Modelfile**：
   ```
   FROM ~/Models/Qwen2.5-14B-Instruct-Q4_K_M.gguf
   PARAMETER num_ctx 32768
   PARAMETER num_gpu 99
   PARAMETER cache_type_k q4_0
   PARAMETER cache_type_v q4_0
   PARAMETER flash_attn true
   PARAMETER mlock true
   PARAMETER num_thread 8
   ```

2. **Ollama 已知支持这些参数**（`docs/modelfile.mdx`）—— 验证 flash_attn 和 cache_type_k/v 在 Ollama 0.30 是否暴露

3. **可能的小 patch**：如果 Ollama 没暴露某些 flag，patch Ollama + PR

4. **fallback**：如果 Ollama 不让 14B 32K 跑（虽然 Phase 4 证明 llama.cpp 可以），用裸 llama.cpp + wrapper script

### 5.3 验收标准

- 老大执行 `curl ... | bash` 后 5 分钟内能 `ollama run qwen2.5-14b-fusion:32k`
- 第一次响应 < 30s（prefill 32K ~6 min 是极端值，正常 8K prompt 30-60s）
- Gen ≥ 5 t/s
- 16GB Air + macOS 26.5.1 + M5 工作
- README 写清楚：是什么 / 为什么 / 怎么用 / 性能数据

### 5.4 真实风险

- **Ollama 0.30 不暴露 cache_type_k/v flag** → 需要 1-2 天 patch Ollama + PR
- **Ollama 0.30 flash_attn 默认是 auto 不一定开** → 验证 + 强制 ON
- **Ollama 不支持 mlock 参数** → wrap 一个 llama-server 用 llama.cpp 直接调
- **14B 32K 在 Ollama 上 OOM**（跟 Phase 4 一样问题） → 用我们测过的 flag 组合强制 ON

### 5.5 备选：B + A 并行

如果老大想同时推进 A + B：

- 我做 A（产品 + README + 推广）
- 老大（或者另一个人）做 B（spec decode 真训练 + sliding window 收尾）
- A 出来后，B 完成后插进 A 作为 v2 升级

但 B 单独做也要 2 周起，可能太长。**我建议先 A 1-2 周，看 demo 效果，再决定 B**。
